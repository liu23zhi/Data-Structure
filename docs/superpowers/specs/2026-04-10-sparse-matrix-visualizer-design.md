# 稀疏矩阵转置可视化网页设计

## 1. 背景与目标

本设计面向仓库中的 `part3/sparse_matrix.c` 实验，将其改造成一个可在浏览器中展示的教学型可视化页面。页面需要把稀疏矩阵转置的相关功能以 HTML 形式呈现给用户，并通过图形化界面详细说明代码运行的每一个步骤。

本次设计的目标是：

- 只覆盖 `part3/sparse_matrix.c`
- 面向老师验收与同学学习两个场景
- 用前端页面忠实复现现有 C 代码的核心逻辑
- 同时支持自动播放与手动单步演示
- 展示普通转置、快速转置、优化快速转置三种算法
- 用清晰的可视化和文字说明解释每一步状态变化

## 2. 设计边界

### 2.1 本次范围

本次实现只包含以下内容：

1. 固定使用 `part3/sparse_matrix.c` 中当前示例矩阵
2. 展示三元组表构建过程
3. 展示普通转置过程
4. 展示快速转置过程
5. 展示优化快速转置过程
6. 展示三种算法结果一致性说明
7. 展示时间复杂度与空间复杂度对比

### 2.2 非目标

以下内容不纳入本次设计：

- 不支持用户自定义输入矩阵
- 不增加后端服务
- 不直接在网页中调用 C 可执行文件
- 不扩展到 `part1` 或 `part2`
- 不做工程化打包或复杂构建流程

这样可以保证页面聚焦于教学展示，而不是变成一个通用算法平台。

## 3. 设计原则

本页面遵循以下原则：

1. **与现有代码一致**：页面展示的算法阶段、变量命名、步骤顺序应尽量贴合 `part3/sparse_matrix.c`
2. **可教学**：用户不仅看到结果，还能理解“为什么这样变化”
3. **可演示**：适合答辩或课堂演示，界面布局清晰，重点突出
4. **可控播放**：用户可以自动播放，也可以逐步前进和回退
5. **最小实现成本**：采用静态 HTML/CSS/JS 文件，不引入额外后端依赖

## 4. 技术方案

采用纯前端静态页面实现，但逻辑层严格按照现有 C 代码重建步骤。

### 4.1 技术选型

- HTML：承载页面结构
- CSS：实现教学展示所需的图形化界面、高亮、布局和状态样式
- JavaScript：负责算法步骤建模、状态推进、播放控制和渲染

### 4.2 方案结论

采用“前端 + C 程序结果映射”的方案：

- 不在浏览器中直接运行 C 代码
- 由 JavaScript 根据 `part3/sparse_matrix.c` 的逻辑生成步骤快照
- 页面展示每个步骤对应的状态变化和解释文字

这样既能做到强交互，也能保证页面与实验代码一一对应。

## 5. 页面信息架构

页面设计为单页教学演示器，包含六个核心区域。

### 5.1 顶部标题区

展示：

- 实验标题
- 页面副标题
- 当前算法标签
- 简短实验说明

目标是让用户进入页面后立即知道这是稀疏矩阵转置实验的可视化教学页。

### 5.2 控制区

提供以下交互控件：

- 算法切换：普通转置 / 快速转置 / 优化快速转置
- 播放 / 暂停
- 上一步 / 下一步
- 重置
- 播放速度切换：慢 / 中 / 快

要求：

- 切换算法后自动回到该算法的第一个步骤
- 自动播放本质上是定时推进步骤
- 单步模式要求可逆，即支持回退到上一状态

### 5.3 数据展示区

用于展示算法输入和输出，包含：

- 原始矩阵二维视图
- 原始三元组表
- 转置结果三元组表
- 转置结果二维矩阵视图

在步骤推进时，这些视图会根据当前快照同步更新，并对当前操作位置进行高亮。

### 5.4 步骤讲解区

用于告诉用户“现在执行到了哪里”。该区域展示：

- 当前步骤标题
- 当前步骤详细说明
- 对应 C 函数名
- 当前关键变量
- 当前步骤编号和总步骤数

讲解文案要求使用人话表述，而不是只显示变量名。

### 5.5 算法状态区

用于展示不同算法特有的运行状态。

- 普通转置：当前外层列扫描位置、当前遍历到的三元组位置、当前写入结果位置
- 快速转置：`num[]`、`cpot[]` 的计算与变化
- 优化快速转置：单数组 `cpot[]` 的统计、转起始位置、递增复用过程

### 5.6 复杂度对比区

展示三种算法的：

- 时间复杂度
- 空间复杂度
- 当前示例参数下的估算操作次数
- 为什么快速转置更高效的简短说明

该区域既可以常驻显示，也可以在演示进入总结阶段时重点高亮。

## 6. 与 C 代码的映射关系

页面逻辑必须明确对应到 `part3/sparse_matrix.c` 中的函数与阶段。

### 6.1 数据构建阶段

对应 `build_tsmatrix`

职责：

- 从固定二维矩阵扫描非零元素
- 生成原始三元组表
- 为后续三种算法提供统一输入

### 6.2 普通转置阶段

对应 `transpose_naive`

页面需表现：

- 外层按列扫描
- 每扫描一列，就遍历全部三元组
- 命中当前列时，将元素写入结果表
- 最终生成转置结果

### 6.3 快速转置阶段

对应 `transpose_fast`

页面需表现：

1. 统计每列非零元素个数，得到 `num[]`
2. 根据前缀和计算 `cpot[]`
3. 遍历原始三元组表
4. 将元素直接放入转置表对应位置
5. 写入后更新 `cpot[]`

### 6.4 优化快速转置阶段

对应 `transpose_fast_optimized`

页面需表现：

1. 先复用 `cpot[]` 统计每列非零元素个数
2. 通过从后向前的累计计算将其转换为起始位置
3. 再遍历原三元组表写入结果
4. 写入后递增同一个 `cpot[]`

重点在于让用户理解“同一个数组承担多个阶段职责”的优化思想。

### 6.5 总结阶段

对应：

- `verify_equal`
- `complexity_analysis`

页面需展示：

- 三种算法结果一致
- 操作次数估算
- 时间复杂度与空间复杂度差异

## 7. 数据模型设计

前端以“步骤快照”驱动页面，而不是边播边临时计算界面状态。

### 7.1 基础数据

初始数据直接来自 `part3/sparse_matrix.c` 当前示例矩阵，即固定 6×6 输入矩阵。

页面启动时先生成：

- 原始二维矩阵
- 原始三元组表
- 三种算法各自的步骤序列

### 7.2 步骤对象

每个步骤对象至少包含以下字段：

- `algorithm`：算法标识
- `phase`：所属阶段
- `title`：步骤标题
- `description`：步骤说明
- `functionName`：对应 C 函数名
- `matrixState`：矩阵展示状态
- `sourceTriples`：原三元组表状态
- `resultTriples`：结果三元组表状态
- `highlight`：当前高亮位置
- `variables`：关键变量快照
- `stepIndex`：当前步骤下标

这样可以支持：

- 自动播放
- 单步前进
- 单步回退
- 切换算法后从头演示

### 7.3 状态生成策略

建议在页面初始化时一次性生成所有步骤，而不是在点击时动态推导。原因如下：

- 上一步操作更容易实现
- 自动播放逻辑更稳定
- 每一步的讲解文案可以提前绑定
- 页面渲染和算法逻辑解耦更清晰

## 8. 交互设计

### 8.1 算法切换

用户可以在三种算法之间切换：

- 普通转置
- 快速转置
- 优化快速转置

切换规则：

- 切换时重置到该算法第 1 步
- 控制区按钮状态同步刷新
- 页面讲解区、状态区、结果区全部同步刷新

### 8.2 播放控制

支持以下行为：

- 播放：按当前速度自动推进步骤
- 暂停：停止推进
- 上一步：回到前一快照
- 下一步：进入后一快照
- 重置：回到当前算法初始状态

### 8.3 速度控制

至少提供三档：

- 慢速：适合课堂讲解
- 中速：适合自学浏览
- 快速：适合完整回放

### 8.4 交互反馈

按钮状态需具备明确反馈，例如：

- 在第一步时禁用“上一步”
- 在最后一步时禁用“下一步”
- 播放中按钮显示“暂停”

## 9. 可视化设计细节

### 9.1 高亮规则

页面使用颜色或边框强调当前步骤相关元素，包括：

- 当前扫描到的矩阵单元格
- 当前扫描到的原三元组项
- 当前写入的结果三元组项
- 当前更新的 `num[]` 或 `cpot[]` 元素
- 当前扫描列

### 9.2 普通转置可视化重点

重点突出：

- “按列扫描”的外层逻辑
- “每列都要重新遍历全部三元组”的成本
- 命中列时才会写入结果

### 9.3 快速转置可视化重点

重点突出：

- `num[]` 是如何统计出来的
- `cpot[]` 如何由前缀和得到
- 元素如何通过 `cpot[col]` 直接定位到结果表位置
- 写入后 `cpot[col]` 为什么要加一

### 9.4 优化快速转置可视化重点

重点突出：

- `cpot[]` 在不同阶段扮演不同角色
- 从后向前计算起始位置的逻辑
- 与普通快速转置相比，为什么辅助空间减少

## 10. 文案与教学表达

页面说明文案应偏教学口吻，例如：

- “现在正在扫描原矩阵的第 3 列”
- “这一列共有 3 个非零元素，因此 `num[3] = 3`”
- “根据前缀和可知，第 3 列转置后应从结果表的第 4 个位置开始写入”
- “当前把 `(row=4, col=3, val=24)` 转换为 `(row=3, col=4, val=24)`”

避免仅显示抽象变量，不做解释。

## 11. 文件组织

本次实现的静态资源放在 `part3/` 下：

- `part3/sparse_matrix_visualizer.html`
- `part3/sparse_matrix_visualizer.css`
- `part3/sparse_matrix_visualizer.js`

职责划分：

- HTML：页面骨架与区域容器
- CSS：布局、配色、状态高亮、表格样式、响应式细节
- JS：数据构建、步骤生成、状态切换、事件绑定、播放逻辑

## 12. 验证方案

实现后至少验证以下内容：

### 12.1 数据正确性

- 原始矩阵与 `part3/sparse_matrix.c` 中示例数据完全一致
- 原始三元组表与 C 代码逻辑一致
- 三种算法最终转置结果一致

### 12.2 交互正确性

- 播放 / 暂停功能正常
- 上一步 / 下一步功能正常
- 算法切换后正确重置
- 重置后回到初始状态

### 12.3 教学可读性

- 每个主要步骤都有对应说明文字
- 页面能够清晰说明变量变化
- 页面能体现三种算法差异
- 总结区能解释为什么快速转置更优

## 13. 风险与取舍

### 13.1 风险

- 如果步骤拆得过细，页面播放时间会偏长
- 如果步骤拆得过粗，教学效果会下降
- 优化快速转置的单数组复用概念相对抽象，可视化不当会影响理解

### 13.2 取舍策略

采用“阶段清晰、步骤适中”的粒度：

- 关键状态变化必须单独成步
- 纯静态说明不必拆成过多帧
- 以教学清晰度优先，而不是追求逐行模拟所有 C 语句

## 14. 交付结论

最终交付物是一个位于 `part3/` 下的静态教学演示页，用于可视化展示 `part3/sparse_matrix.c` 中的稀疏矩阵转置过程。该页面将：

- 覆盖普通转置、快速转置、优化快速转置
- 同时支持自动播放与手动单步
- 用图形化界面展示矩阵、三元组表、辅助数组和结果状态
- 用详细文字解释每一步的代码运行过程
- 保持与现有 C 代码逻辑一致，便于教学、验收和答辩展示
